3.1 Software Development Framework

Chapter 3 frames CLOIE development as a structured yet flexible process: tasks are visualized, prioritized continuously, and refined through feedback while staying aligned with capstone milestones.

Backlog

All planned features, requirements, and refinements are listed and prioritized for delivery.

To Do

Ready tasks are selected based on priority, dependencies, and milestone relevance.

In Progress

Active implementation focuses on core modules before layered enhancements.

Testing

Built tasks are validated for correctness, usability, and alignment with expected behavior.

Done

Verified tasks are marked complete and retained as stable increments.

Why Kanban Was Selected

The capstone documentation highlights Kanban's fit for a small team with evolving requirements. It supports visible progress, continuous reprioritization, and incremental delivery without sprint overhead while still preserving milestone discipline.

3.2 Requirements

The requirements specification is organized into user expectations, required system behaviors, and quality constraints that keep CLOIE practical, secure, and maintainable for real academic use.

3.2.1 User Requirements

CLOIE supports role-specific workflows for all intended stakeholder groups:

Faculty Students Graduating Students Alumni Industry Partners Program Heads and Deans System Administrators
  • Stakeholders require secure, role-based access and context-specific outputs.
  • Users need clear forms, confidentiality protection, and efficient participation across devices.
  • Program-level customization must remain controlled and isolated by design.

3.2.2 Functional Requirements

The system must deliver core end-to-end modules for evaluation and reporting:

  • User and access management
  • Outcomes management (PLOs, GOs, and course-level mappings)
  • Evaluation instrument management with version control
  • Evaluation cycle configuration and control
  • Response collection, validation, and processing
  • Analytics and consolidated reporting
  • Qualitative feedback handling (including optional prompt-assisted input)
  • Administrative configuration and governance controls

3.2.3 Non-Functional Requirements

Quality standards ensure the platform remains dependable and usable in institutional settings:

Usability Performance Security Confidentiality Reliability Scalability Availability Responsiveness Maintainability

These constraints align with CLOIE's intended role as a multi-user, report-oriented, college-level monitoring system.

3.2.4 Software and Hardware Requirements

Software Requirements

Development Software

  • Windows 10/11, Visual Studio Code, Git/GitHub
  • Figma and modeling tools (Visual Paradigm/Draw.io)
  • Browser testing via Chrome, Firefox, Edge, or Safari

Application Development Stack

  • Next.js and TypeScript
  • Tailwind CSS and shadcn/ui
  • React Hook Form, Zod, TanStack Query
  • Chart.js, Recharts, wordcloud2.js, winkNLP, stopword
  • PostgreSQL and Supabase
  • Vitest and Playwright

Deployment Services

  • Cloud hosting suitable for web/PWA deployment
  • Vercel-equivalent frontend deployment
  • Supabase-equivalent managed backend/database services

Hardware Requirements

For Development

  • Intel Core i5 / Ryzen 5 or higher
  • At least 8 GB RAM
  • At least 256 GB SSD
  • Stable broadband internet connection

For End Users

  • Desktop, laptop, tablet, or smartphone
  • Modern web browser
  • Stable internet access for participation and submission

This hardware profile supports cross-device access for students, faculty, alumni, industry partners, and administrators.

3.3 Feasibility Issues

Feasibility analysis in Chapter 3 evaluates whether CLOIE is practical to build and adopt within capstone constraints and institutional context.

3.3.1 Economic Feasibility

The proposed web/PWA direction is financially realistic for a student-led project because it avoids separate native implementations and leverages widely available tools and cloud services.

3.3.2 Technical Feasibility

CLOIE's required modules fit modern web architecture and relational data structures, with incremental delivery possible from core modules to advanced reporting features.

3.3.3 Operational Feasibility

The system aligns with existing academic practices at Assumption College of Davao and complements current institutional systems rather than replacing LMS/SIS workflows.

3.3.4 Schedule Feasibility

The March to September 2026 timeline is workable through staged milestones, early core-module prioritization, and continuous Kanban-guided progress.
